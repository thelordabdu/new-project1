## Results

**Our rolling RMSSD (awake, 86 bpm):** ~35–55 ms
**Whoop app HRV (overnight):** 72 ms
**Gap:** ~25–50%

This gap is expected and not a bug. Whoop computes HRV overnight during deep sleep when HR is at its lowest (typically 50–60 bpm) and the user is completely still. We're computing in real-time while awake at 86 bpm. Beat-to-beat variance is naturally lower when awake.

Both values are in the same physiological range (20–80 ms resting HRV). Order of magnitude matches. v0.1 validation passes.

A true apples-to-apples comparison requires running BLE collection overnight — not practical for a browser app. This is a hardware/iOS problem deferred to v1.

## Bug found and fixed: artifact filter ineffective on small packets

**Problem:** Values like 327, 331, 345, 378 ms were passing `cleanRR()` even though they imply HR >180 bpm, which is impossible at rest.

**Root cause:** `filterArtifacts()` computes the local median from within the current packet only. With 1–2 values per packet, the median IS the suspicious value — so it always passes the 20% deviation check. The filter only works correctly with larger arrays.

**Fix applied:** Raised the physiological floor in `filterPhysiological()` from 273 ms (HR 220) to 333 ms (HR 180).

```typescript
// Before
return rr.filter(v => v >= 273 && v <= 2000)

// After
return rr.filter(v => v >= 333 && v <= 2000)
```

**Note for v0.6 (workout mode):** During high-intensity exercise, HR can legitimately exceed 180 bpm, which would produce valid R-R intervals below 333 ms. The physiological floor will need to be revisited when workout mode is built — likely made context-aware (resting vs active) or lowered back to 273 ms with a better cross-packet artifact filter replacing the simple median check.
