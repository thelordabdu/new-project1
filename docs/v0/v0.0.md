# v0.0 — BLE Proof of Concept

**Date:** 2026-05-10  
**Device:** WHOOP 5A00461224 (Whoop 5)  
**Test page:** `app/(app)/test-ble/page.tsx` at `localhost:3000/test-ble`

---

## What v0.0 does

- [x] Connects to Whoop over Web Bluetooth (Chrome only)
- [x] Reads Heart Rate Service (UUID `0x180D`) + HR Measurement characteristic (UUID `0x2A37`)
- [x] Parses HR value from BLE packet flags (8-bit and 16-bit format handled)
- [x] Parses R-R intervals from the same packet when bit 4 of flags is set
- [x] Converts raw R-R units to milliseconds (`raw / 1024 * 1000`)
- [x] Buffers last 100 samples in React state
- [x] Computes live RMSSD across all buffered R-R intervals
- [x] Displays HR, latest R-R intervals, RMSSD, and raw sample log in the browser
- [x] Logs every packet to browser console

---

## Findings

**Device broadcasts R-R intervals:** Yes — bit 4 of flags byte is set, R-R data present in every packet.

**Packets per second:** ~1 packet/sec (consistent with standard BLE HR profile notification rate).

**R-R intervals per packet:** 1–3 intervals per packet. Example:
```json
{ "ts": 1778399728298, "hr": 88, "rr_intervals": [571, 702, 686] }
{ "ts": 1778399729317, "hr": 88, "rr_intervals": [707] }
```

**HR observed:** 83–88 bpm at rest. Physiologically valid.

**R-R range observed:** 571–1271 ms. Mostly valid (normal range 273–2000 ms). Occasional spikes (e.g. 1271 ms followed by 594 ms in the same packet) — likely motion artifacts or brief contact loss.

**RMSSD observed:** 260–398 ms — significantly inflated. Root cause: RMSSD is computed across all buffered R-R values including across packet boundaries, treating the last interval of one packet and the first interval of the next as consecutive beats. They may not be. Inter-packet diffs are noise, not signal.

**HR vs Whoop app:** Not formally compared this session. Visually consistent (~87 bpm on page vs expected resting HR). Formal comparison needed in v0.1.

---

## Known issues

**RMSSD is wrong.** The current implementation flattens all R-R intervals from all packets into one array and diffs sequentially. This means:
1. Inter-packet boundaries produce spurious diffs
2. Motion artifacts (e.g. 1271 ms spike) are not filtered before computing

This is a v0.0 limitation — acceptable for proof of concept, not for any real HRV use.

---

## Suggested improvements for v0.1

These are in addition to what's already planned (artifact filtering via `cleanRR()`, proper RMSSD per-packet):

- **Packet timestamp gap detection** — if `ts` delta between two consecutive samples is >2s, treat them as non-contiguous. Never diff R-R across a gap.
- **Signal quality indicator** — show a simple "signal quality" score based on % of R-R values that passed the physiological filter (273–2000 ms). If <80% pass, warn the user.
- **Session export button** — one-click copy/download of the raw sample buffer as JSON. Right now you have to manually copy from the `<pre>` block.
- **Running HR zone display** — show which HR zone (1–5) the user is currently in, computed from latest HR. Useful visual sanity check during a workout.
- **Per-packet RMSSD** — compute RMSSD only within each packet's R-R array (not across packets). Display a rolling average of per-packet RMSSD values instead.
- **Reconnect on disconnect** — Whoop BLE drops occasionally. Auto-attempt reconnect once before requiring manual action.
